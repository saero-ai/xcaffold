# Project Layouts

xcaffold discovers resources by scanning the `xcf/` directory recursively for `*.xcf` files. Every file is routed by its `kind:` field — there is no filename convention required and no manifest that must be kept in sync. The filesystem is the source of truth.

## Filesystem Boundaries

Before choosing a layout, understand the three hard boundaries xcaffold enforces:

1. **Source files** (`xcf/`) — User-authored `.xcf` resource definitions. Scanned recursively. Version-controlled.
2. **State files** (`.xcaffold/`) — Tool-managed, gitignored. Contains `*.xcf.state` files written by `xcaffold apply`. Never edit these manually.
3. **Compiled output** (`.claude/`, `.cursor/`, `.gemini/`, `.agents/`, `.github/`) — Generated by `xcaffold apply`. Do not edit these directories directly; edits are overwritten on the next apply and will be flagged as drift by `xcaffold status`.

> [!IMPORTANT]
> `kind: project` and all resource files live in `xcf/` (or at the project root). The `.xcaffold/` directory is tool-managed only — it holds state files, not source files. A `project.xcf` at `.xcaffold/project.xcf` is an auto-generated snapshot, not the authoritative source.

## Layout 1 — Single File (Tutorial / Personal)

**Best for:** Personal projects, quick experiments, and CLI tutorials with 1–3 agents.

All resources in a single file:

```
my-project/
├── project.xcf          ← kind: project (metadata + targets)
├── agent.xcf            ← kind: agent (inline)
└── .claude/             ← generated output
```

`project.xcf` (pure YAML, no frontmatter delimiters):

```yaml
kind: project
version: "1.0"
name: my-project
targets:
  - claude
```

`agent.xcf` (frontmatter + body):

```
---
kind: agent
version: "1.0"
name: developer
model: sonnet-4
---
You are a generalist developer. Help with code reviews and debugging.
```

## Layout 2 — Organized by Resource Type

**Best for:** Small-to-medium projects (up to ~5 agents) where one developer manages everything.

Separate each resource kind into its own directory under `xcf/`:

```
my-project/
├── project.xcf
└── xcf/
    ├── agents/
    │   └── developer/
    │       └── agent.xcf
    ├── skills/
    │   └── git-workflow/
    │       └── skill.xcf
    └── rules/
        └── testing/
            └── rule.xcf
```

> [!NOTE]
> Agents and skills **must** live in a subdirectory under their respective kind folder. A file at `xcf/agents/developer.xcf` (flat) causes a parse error. The correct path is `xcf/agents/developer/agent.xcf`. All resource kinds support directory-per-resource layout with canonical filenames (`agent.xcf`, `skill.xcf`, `rule.xcf`, `workflow.xcf`, `mcp.xcf`).

## Layout 3 — Organized by Domain

**Best for:** Large, cross-functional projects with multiple teams and CODEOWNERS rules.

Group resources by the domain they govern. The scanner discovers all `*.xcf` files recursively regardless of nesting depth, so domain-based grouping is simply a naming convention — xcaffold imposes no restrictions on how you nest directories under `xcf/`:

```
my-project/
├── project.xcf
└── xcf/
    ├── frontend/
    │   ├── agents/
    │   │   └── ui-developer/
    │   │       └── agent.xcf
    │   └── rules/
    │       └── accessibility/
    │           └── rule.xcf
    ├── backend/
    │   ├── agents/
    │   │   └── api-developer/
    │   │       └── agent.xcf
    │   └── rules/
    │       └── api-conventions/
    │           └── rule.xcf
    ├── policies/
    │   └── require-descriptions.xcf
    └── blueprints/
        ├── frontend.xcf
        └── backend.xcf
```

With this layout, `CODEOWNERS` rules naturally map to xcaffold resource ownership:

```
xcf/frontend/ @frontend-team
xcf/backend/  @backend-team
xcf/policies/ @platform-team
```

## When to Introduce Blueprints

Once you have a domain-split layout with multiple teams, `kind: blueprint` files let each team compile only their subset of agents and rules. See [Blueprint Design](blueprint-design) for guidance on when and how to introduce them.

## Key Rules at a Glance

| Rule | Why |
|---|---|
| Agents use `xcf/agents/<name>/agent.xcf` | Canonical filename; parser rejects flat agent files |
| Skills use `xcf/skills/<name>/skill.xcf` | Canonical filename; parser rejects flat skill files |
| Duplicate resource IDs across files are rejected | `mergeAllStrict` enforces uniqueness — reorganize, don't rename |
| `.xcaffold/` is gitignored and tool-managed | State files are not source files |
| Never edit compiled output directories | `xcaffold status` flags manual edits as drift |
