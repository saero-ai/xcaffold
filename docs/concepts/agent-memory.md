# Agent Memory

Memory entries are durable, agent-scoped contexts that persist across sessions. Each memory entry is a plain `.md` file placed under `xcf/agents/<agent-id>/memory/`. When you run `xcaffold apply`, these files are compiled into each target provider's native persistence mechanism.

For providers with no first-class memory format, xcaffold emits a `MEMORY_NO_NATIVE_TARGET` fidelity note. See [Provider Support](#provider-support) for the current status per target.

---

## Example Usage

Place each memory file under `xcf/agents/<agent-id>/memory/`:

```
xcf/
  agents/
    security-reviewer/
      security-reviewer.xcf
      memory/
        audit-log-owner.md
```

`xcf/agents/security-reviewer/memory/audit-log-owner.md`:

```markdown
The security team owns the audit log. Route all audit-log
questions to them rather than the feature team.
```

Directory placement encodes agent affiliation. A file at `xcf/agents/<agent-id>/memory/<name>.md` binds the memory entry to `<agent-id>`. At compile time the compiler sets `AgentRef` to the agent directory that contains the `memory/` subdirectory.

### Optional Frontmatter

Memory files can include optional YAML frontmatter with `name` and `description`:

```markdown
---
name: Audit Log Owner
description: Who owns the project audit log.
---
The security team owns the audit log. Route all audit-log
questions to them rather than the feature team.
```

When frontmatter is absent, the filename (without `.md`) is used as the name, and the first line of the file is used as the description.

---

## File Format

Memory files are plain Markdown. There is no `kind:`, `version:`, or `instructions:` field — memory is not a `.xcf` resource kind.

| Element | Required | Description |
|---|---|---|
| Frontmatter `name` | no | Display name. Defaults to filename stem. |
| Frontmatter `description` | no | Summary. Defaults to first line of content. |
| Body | yes | Markdown content loaded as the memory entry. |

---

## Compiled Output

After `xcaffold apply`, each agent's memory produces:

| File | Content |
|---|---|
| `<agent-dir>/MEMORY.md` | Link-list index: `- [Name](file.md) — Description` |
| `<agent-dir>/<name>.md` | Individual memory file content |

For Claude Code, output lands in `.claude/agent-memory/<agent-id>/`.

MEMORY.md is auto-generated — do not edit it directly. Edit the source `.md` files under `xcf/agents/<agent-id>/memory/` instead.

---

## Provider Support

| Provider | Status | Behavior |
|---|---|---|
| Claude Code | Supported | Writes `MEMORY.md` index + individual `.md` files under `.claude/agent-memory/<agent-id>/`. |
| Gemini CLI | Partial | Appends entries to `GEMINI.md` under a `## Gemini Added Memories` section with marker comments. Emits `MEMORY_PARTIAL_FIDELITY` fidelity note. |
| Antigravity | Supported | Writes `knowledge/<name>.md` files with YAML frontmatter tags. |
| Cursor | Not supported | No native cross-session memory. Emits `MEMORY_NO_NATIVE_TARGET` fidelity note. |
| GitHub Copilot | Not supported | No native cross-session memory. Emits `MEMORY_NO_NATIVE_TARGET` fidelity note. |

---

## Import

`xcaffold import` discovers existing memory files from provider directories and copies them directly into `xcf/agents/<agent-id>/memory/`:

```bash
xcaffold import --provider claude
```

MEMORY.md index files are skipped during import (they are auto-generated by `apply`). If a memory directory references an agent that was not found during import, a warning is emitted but the memory is still imported.
