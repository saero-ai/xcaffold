# Getting Started with xcaffold

This tutorial walks through creating your first agent configuration with xcaffold. By the end, you will have a working `scaffold.xcf` file compiled to your target platform's native format.

**Time to complete:** ~10 minutes  
**Prerequisites:** xcaffold installed and available on `$PATH`, any text editor, an empty project directory.

No AI subscription or API key is required to run `init` and `apply`.

---

## Step 1 — Initialize a new project

Open a terminal in your empty project directory and run:

```bash
xcaffold init --yes
```

The `--yes` flag accepts all defaults non-interactively. xcaffold infers the project name from the directory name — it never prompts for one.

**Expected output:**

```
  xcaffold v0.x.x
  The deterministic agent configuration compiler.
  ──────────────────────────────────────────────────

  Welcome! Let's scaffold your agents.

  Created xcf/references/ — field reference for resource kinds

✓ Created scaffold.xcf
  Project: my-project | Targets: claude
  Starter agent: developer (model: claude-sonnet-4-6)

  Edit your agents, then run 'xcaffold apply'.
```

xcaffold detects the first available agent runtime on your `$PATH` and uses it as the suggested default target. All five targets — `claude`, `cursor`, `antigravity`, `copilot`, and `gemini` — are fully supported and selected via `--target` on apply. 

The `init` command generates a scalable multi-file directory layout for your project. This prevents configuration bloat as your agent ecosystem grows:

```
my-project/
  scaffold.xcf              # kind: project
  xcf/
    agents/
      developer.xcf         # kind: agent
    policies/
      safety.xcf            # kind: policy
    rules/
      conventions.xcf       # kind: rule
    settings.xcf            # kind: settings
```

Open `scaffold.xcf` in your editor. It acts as the project manifest, linking your targeted platforms with your resource documents:

```yaml
# scaffold.xcf — generated by xcaffold init
kind: project
version: "1.0"
name: "my-project"
targets:
  - claude
agents:
  - developer
rules:
  - conventions
policies:
  - require-agent-description
  - require-agent-instructions

# Optional: Configure the 'xcaffold test' simulator.
# test:
#   cli-path: "claude"   # Path to CLI binary. Defaults to 'claude' on $PATH.
#   judge-model: "claude-sonnet-4-6" # Model used for --judge evaluation.
```

Your actual agent definitions are scaffolded securely inside `xcf/agents/developer.xcf`.

This structure is the source of truth. Every file xcaffold produces traces back to this `xcf/` configuration layout. You never edit generated output directly.

---

## Step 2 — Compile to your target

Run apply with an explicit target. This example uses `claude` (the default):

```bash
xcaffold apply --target claude
```

The `claude` target compiles your configuration into `.claude/` — the native Claude Code directory format with YAML-frontmatter agent files and `settings.json`.

**Terminal output:**

```
  [project] ✓ wrote /path/to/my-project/.claude/agents/developer.md  (sha256:<hex>)
  [project] ✓ wrote /path/to/my-project/.claude/settings.json  (sha256:<hex>)

[project] ✓ Apply complete. scaffold.claude.lock updated.
```

To target Gemini CLI instead:

```bash
xcaffold apply --target gemini
```

Inspect the result:

```bash
cat .claude/agents/developer.md
```

---

## Step 3 — Inspect the compiled output

`.claude/agents/developer.md` contains exactly what `scaffold.xcf` described, in YAML frontmatter + Markdown body format:

```markdown
---
name: developer
description: General software developer agent.
model: claude-sonnet-4-6
effort: high
tools:
  - Bash
  - Read
  - Write
  - Edit
  - Glob
  - Grep
---

You are a software developer.
Write clean, maintainable code.
```

Every line maps back to `scaffold.xcf`:

| Output field | Source field |
|---|---|
| `name: developer` | agents key: `developer` |
| `model: claude-sonnet-4-6` | `model: "claude-sonnet-4-6"` |
| `description:` | `description:` |
| `effort: high` | `effort: "high"` |
| `tools:` list | `tools:` array |
| Body text | `instructions:` block |

> **Note on fidelity:** Different targets support different fields. The `claude` target preserves all fields. Targets with a narrower feature set (such as `gemini` or `antigravity`) drop fields they have no equivalent for and emit a warning for each one. Use `xcaffold apply --check-permissions --target <target>` to preview which fields will be dropped before compiling.

## Step 4 — Examine the lock file

`xcaffold apply` writes a lock file alongside your configuration. For `--target claude`:

```
scaffold.claude.lock
```

Its content:

```yaml
last_applied: "2026-04-11T10:00:00Z"
xcaffold_version: "0.x.x"
claude_schema_version: "alpha"
target: "claude"
scope: "project"
config_directory: "."
source_files:
  - path: scaffold.xcf
    hash: "sha256:abc123..."
artifacts:
  - path: .claude/agents/developer.md
    hash: "sha256:def456..."
  - path: .claude/settings.json
    hash: "sha256:ghi789..."
version: 2
```

The lock file records SHA-256 hashes of both the source (`scaffold.xcf`) and the compiled output files. Each target has its own lock file (`scaffold.gemini.lock`, `scaffold.cursor.lock`, etc.) so multiple targets can be maintained simultaneously without invalidating each other.

**What this enables:**

- `xcaffold apply` skips recompilation when source hashes are unchanged:

  ```
  [project] Sources unchanged — skipping compilation. Use --force to recompile.
  ```

- `xcaffold diff` compares the current file hashes against the lock to detect manual edits. Any change to compiled output made outside xcaffold registers as drift and blocks the next apply unless you use `--force`.

Lock files are committed to version control alongside `scaffold.xcf`. Together they give you a reproducible, auditable agent configuration.

---

## Step 5 — Make a change and re-apply

Open `scaffold.xcf` and update the instructions:

```yaml
    instructions: |
      You are a software developer.
      Write clean, maintainable code.
      Always run tests before marking a task complete.
```

Run apply again:

```bash
xcaffold apply --target claude
```

```
  [project] ✓ wrote /path/to/my-project/.claude/agents/developer.md  (sha256:<hex>)

[project] ✓ Apply complete. scaffold.claude.lock updated.
```

The new instruction line appears in `.claude/agents/developer.md`. The lock's artifact hash updates. The source `scaffold.xcf` remains the only file you edited.

## What You Built

You created a `scaffold.xcf` file with a project manifest and one agent definition, compiled it to `.claude/agents/developer.md` using the `claude` target, and observed how the lock file records SHA-256 hashes of both source and output. You verified that xcaffold recompiles only when the source changes, and that the lock prevents unrecorded drift on subsequent applies.
