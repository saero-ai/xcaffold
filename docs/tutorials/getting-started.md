# Getting Started with xcaffold

This tutorial walks through creating your first agent configuration with xcaffold. By the end, you will have a working `scaffold.xcf` file compiled to `AGENTS.md` — a portable, plain-Markdown format that any agent runtime can read.

**Time to complete:** ~10 minutes  
**Prerequisites:** xcaffold installed and available on `$PATH`, any text editor, an empty project directory.

No AI subscription or API key is required to run `init` and `apply`.

---

## Step 1: Initialize a new project

Open a terminal in your empty project directory and run:

```bash
xcaffold init --yes
```

The `--yes` flag accepts all defaults non-interactively. xcaffold infers the project name from the directory name — it never prompts for one.

**Terminal output:**

```
  xcaffold v0.x.x
  The deterministic agent configuration compiler.
  ──────────────────────────────────────────────────

  Welcome! Let's scaffold your agents.

✓ Created scaffold.xcf
  Project: my-project | Target: claude
  Starter agent: developer (model: claude-sonnet-4-6)

  Edit your agents, then run 'xcaffold apply'.
```

xcaffold detects the first available agent runtime on your `$PATH` and uses it as the suggested default target — `claude`, `cursor`, or `antigravity`. Other targets (including `agentsmd`, the most portable format) are fully supported and selected via `--target` on apply.

Open `scaffold.xcf` in your editor. It uses multi-kind format — a `kind: project` document for project metadata, followed by individual resource documents separated by `---`:

```yaml
# scaffold.xcf — generated by xcaffold init
kind: project
version: "1.0"
name: "my-project"
targets:
  - claude

# Optional: Configure the 'xcaffold test' simulator.
# test:
#   cli_path: "claude"   # Path to CLI binary. Defaults to 'claude' on $PATH.
#   judge_model: "claude-sonnet-4-6" # Model used for --judge evaluation.

---
kind: agent
version: "1.0"
name: developer
description: "General software developer agent."
instructions: |
  You are a software developer.
  Write clean, maintainable code.
model: "claude-sonnet-4-6"
effort: "high"
tools: [Bash, Read, Write, Edit, Glob, Grep]

# Assertions are evaluated by the LLM-as-a-Judge when running
# 'xcaffold test --judge'. Define expected behavioral constraints here.
# assertions:
#   - "The agent must not write files outside the project directory."
#   - "The agent must run tests before marking a task complete."
```

The `kind: project` document declares project metadata and targets. Each resource (agents, skills, rules) is a separate `kind: <type>` document in the same file. This is the **multi-kind format** — xcaffold's primary configuration style.

This file is the source of truth. Every file xcaffold produces traces back to it. You never edit generated output directly.

> **Legacy format:** If you have an older `kind: config` file with resources nested under `project:`, it continues to work. `kind: project` is the current recommended format — it supports both single-file and split-file layouts. See [Split Configurations](../how-to/split-configs.md) for details.

---

## Step 2: Compile to AGENTS.md

Run apply with an explicit target:

```bash
xcaffold apply --target agentsmd
```

The `agentsmd` target compiles your configuration to a single `AGENTS.md` at the project root — plain Markdown, no vendor-specific schema.

**Terminal output:**

```
  [project] ✓ wrote /path/to/my-project/AGENTS.md  (sha256:<hex>)

[project] ✓ Apply complete. scaffold.agentsmd.lock updated.
```

Inspect the result:

```bash
cat AGENTS.md
```

---

## Step 3: The compiled output

`AGENTS.md` contains exactly what `scaffold.xcf` described:

```markdown
<!-- Generated by xcaffold. Do not edit manually. -->

## Project

my-project

## Agents

### developer
**Model**: claude-sonnet-4-6
**Description**: General software developer agent.

You are a software developer.
Write clean, maintainable code.
```

Every line maps back to `scaffold.xcf`:

| Output line | Source field |
|---|---|
| `### developer` | agents key: `developer` |
| `**Model**: claude-sonnet-4-6` | `model: "claude-sonnet-4-6"` |
| `**Description**: ...` | `description:` |
| Body text | `instructions:` block |

The header comment is the compiler's own assertion that this file must not be hand-edited. xcaffold will overwrite it on the next `apply`.

> **Note on fidelity:** The `agentsmd` format is plain Markdown. Some xcaffold fields — `tools`, `effort`, `skills` — have no AGENTS.md equivalent and are dropped during compilation. xcaffold emits a warning for each dropped field. Use the `claude` target to preserve all fields.

## Step 4: The lock file

`xcaffold apply` writes a lock file alongside your configuration:

```
scaffold.agentsmd.lock
```

Its content:

```yaml
last_applied: "2026-04-11T10:00:00Z"
xcaffold_version: "0.x.x"
claude_schema_version: "alpha"
target: "agentsmd"
scope: "project"
config_directory: "."
source_files:
  - path: scaffold.xcf
    hash: "sha256:abc123..."
artifacts:
  - path: AGENTS.md
    hash: "sha256:def456..."
version: 2
```

The lock file records SHA-256 hashes of both the source (`scaffold.xcf`) and the compiled output (`AGENTS.md`).

**What this enables:**

- `xcaffold apply` skips recompilation when source hashes are unchanged:

  ```
  [project] Sources unchanged — skipping compilation. Use --force to recompile.
  ```

- `xcaffold diff` compares the current file hashes against the lock to detect manual edits. Any change to `AGENTS.md` made outside xcaffold registers as drift and blocks the next apply unless you use `--force`.

Lock files are committed to version control alongside `scaffold.xcf`. Together they give you a reproducible, auditable agent configuration.

---

## Step 5: Make a change and re-apply

Open `scaffold.xcf` and change `effort` from `"high"` to `"low"`:

```yaml
    effort: "low"
```

Run apply again:

```bash
xcaffold apply --target agentsmd
```

Because the `effort` field has no AGENTS.md equivalent, the compiled output is unchanged. xcaffold recompiles (source hash changed) but emits the same `AGENTS.md`. The lock's `source_files` hash updates to reflect the new `scaffold.xcf` content.

To see a change that does affect output, update the instructions instead:

```yaml
    instructions: |
      You are a software developer.
      Write clean, maintainable code.
      Always run tests before marking a task complete.
```

Re-run apply:

```bash
xcaffold apply --target agentsmd
```

```
  [project] ✓ wrote /path/to/my-project/AGENTS.md  (sha256:<hex>)

[project] ✓ Apply complete. scaffold.agentsmd.lock updated.
```

The new instruction line appears in `AGENTS.md`. The lock's artifact hash updates. The source `scaffold.xcf` remains the only file you edited.
